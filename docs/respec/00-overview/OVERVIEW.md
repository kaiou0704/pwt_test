# システム概要：Re:Spec

## 1. システムの目的
Re:Specは、AI駆動開発（AIDD）時代における仕様書管理プラットフォームである。従来のウォーターフォール開発で作成されたドキュメントと、現代のAI駆動開発環境を構造化された仕様書（スペック）で接続し、システム開発のライフサイクル全体を最適化することを目的とする。

## 2. スコープ
本システムは、以下の領域をカバーするプラットフォームとして機能する。
*   **仕様書の構造化・変換**: Notion PRD、Word、PDF、GitHubリポジトリ等の多様なソースをAI可読な構造化データへ変換。
*   **AIインスペクション**: 8観点に基づくAIによる品質スコアリングおよび監査証跡の記録。
*   **同期・更新**: コード変更を検知し、仕様書を自動更新。Notion、Slack、Teams等への乖離アラート通知。
*   **AIDD連携**: 構造化された仕様書を基にしたAI駆動開発環境への橋渡し。

## 3. 用語定義
*   **AIDD (AI Driven Development)**: AIを活用した開発手法。
*   **FDE (Forward Deployed Engineer)**: 大企業市場開拓のための導入支援エンジニア。
*   **TAM/SAM/SOM**: 市場機会の規模を示す指標。
    *   TAM: ¥2.4T+（日本のSW開発工程費およびNotion有料顧客の需要）
    *   SAM: ¥200B（AI活用可能な仕様書管理市場）
    *   SOM: ¥5B（初期3年間のターゲット市場）
*   **構造化データ**: AIが読み取り・解析可能な形式に変換された仕様情報。

## 4. 全体構成

### 4.1 システムアーキテクチャ
各機能はマイクロサービス化されており、疎結合な構成となっている。

*   **入力ソース**: Notion PRD、Word、PDF、GitHub Repo
*   **コアエンジン**:
    *   変換エンジン（構造化）
    *   検査エンジン（8観点AIインスペクション）
    *   同期エンジン（コード変更検知・仕様書自動更新）
*   **出力・連携先**: Word、PDF、SPEC_template、監査証跡JSON、Notionステータス更新、Slack/Teams通知
*   **外部連携**: GitHub（リポジトリ解析および仕様書データのPush）、ローカルAIDD環境（Claude Code等）

### 4.2 主要機能
*   **仕様書の自動変換**: 古い仕様書（Word/PDF等）をAI可読な構造化データへ変換。
*   **AI品質レビュー**: 8観点AIインスペクションによる仕様書の漏れ指摘および修正。
*   **GitHub連携**: 構造化データのGitHubへのプッシュおよびコード変更検知による仕様書自動更新。
*   **リポジトリ解析**: 既存コードから仕様書を自動生成。
*   **通知機能**: 仕様と実装の乖離をNotion/Slack/Teamsへアラート通知。
*   **マイクロサービス設計**: 各機能は疎結合なマイクロサービスとして構成。

### 4.3 ライフサイクルプロセス
1.  **変換**: 各種ドキュメントおよびコードを構造化データへ変換。
2.  **検査**: AIによる品質レビューおよび漏れの指摘・修正。
3.  **同期**: コード変更と仕様書の自動同期。
4.  **通知**: 乖離発生時のアラート通知。

### 4.4 戦略的ポジショニング
Re:Specは、既存の仕様書管理ツールやDeepWiki等のコード逆生成ツールに対し、以下の優位性を持つ。
*   **Notion連携**: Notion PRDの品質スコアリングをフックとしたフリーミアムモデルの採用。
*   **マルチフォーマット対応**: Word等の従来型ドキュメントとNotion等のモダンなツール双方をカバー。
*   **監査対応**: 監査証跡の自動記録による信頼性の担保。
*   **プラットフォーム性**: 古いシステムとAI駆動開発環境を仲介するHubとしての機能。

### 4.5 ビジネスモデル
*   **SaaS**: フリーミアムモデルを採用（Free/Starter/Business/Enterprise）。
*   **Professional Services**: FDEによる導入支援（Quick Start/Enterprise導入/Strategic Partner）。

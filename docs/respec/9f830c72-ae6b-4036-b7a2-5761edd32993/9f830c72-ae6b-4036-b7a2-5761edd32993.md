# AI 駆動開発時代の仕様書（スペック）管理プラットフォーム。

# 古いウォーターフォール開発時代のシステムと、AI 駆動開発したシステムを、構造化した仕様書（スペック）でつなぎ合わせる Hub かつプラットフォーム。

## このシステムがあると何がうれしいのか

*   PDF など古いタイプの仕様を、AI が読めるような構造化ファイルに変換してくれるので、そのまま Claude Code などに読み込めば AI 駆動開発ができるようになる。古いSIer（NTTデータとか富士通とか）は大喜び
*   すでに AI 駆動開発しているユーザーも、既存の仕様書を AI が精査してくれて、人間が読みやすい仕様書（Word 形式など）に変換してくれるので社内共有ドキュメントにできる。古いSIerの中で比較的新しい取り組みをしている中堅PMが大喜び
*   仕様書がないシステムでも、GitHub リポジトリを読み込めば、自動的に仕様書を作成してくれる。仕様書を書くのが嫌いなPMも大喜び

# 古いシステム ⇔ 構造化された仕様書（スペック） ⇔ AI 駆動開発を取りもってくれる便利アプリ
こういうシステムはまだポジションを確立していないので、今から参入すれば競合がいないので売れる＆儲かる！（といいなぁ）

## 主要な機能

| 機能 | 説明 |
| :--- | :--- |
| 1 仕様書の自動変換 | 古い仕様書を input すると、AI 可読用の構造化データに自動変換する |
| 2 AI による品質レビュー | 古い仕様書は質が悪いので、仕様書の漏れを AI が指摘＆修正してくれる |
| 3 GitHub 連携 | 修正＆構造化された仕様書データは GitHub に push できる |
| 4 AI 駆動開発への橋渡し | GitHub のリポジトリを使って、そのまま AI 駆動開発ができる |
| 5 リポジトリ解析 | 仕様書がないシステムについて、既存のリポジトリコードを解析して AI が自動的に仕様書を作成する |
| 6 3 者連携 | ローカルの AIDD 環境（Claude Code + Git） ⇔ GitHub ⇔ Re:Spec という関係 |
| 7 マイクロサービス設計 | 各機能はマイクロサービス化しており疎結合。各機能ごとに開発可能で、開発規模が小さくて済む |

## 技術的な課題と検討事項

### 課題 1: 古い仕様書の構造化データ変換
**問題:** 古い仕様書（Word / Excel / PDF）→ AI ready な構造化データ（Markdown / JSON / PlantUML / Mermaid / YAML）に変換するには、標準ライブラリだけでは無理で、Computer Vision などの機械学習技術が必要かもしれない。

**現状の対応:**

| 形式 | ライブラリ | 処理内容 |
| :--- | :--- | :--- |
| PDF | pdf-parse | テキスト抽出 → Claude API で Markdown 構造化 |
| DOCX | mammoth | HTML 変換 → Markdown 変換 |
| XLSX | xlsx | セルデータ抽出 → テーブル Markdown 化 |
| PPTX | pptxgenjs | XML パースでスライド抽出 |

**今後の改善案:**
*   スキャン PDF や画像埋め込みドキュメントには OCR (Azure AI Vision / Tesseract) が必要
*   レイアウト解析には Azure Document Intelligence (旧 Form Recognizer) が有効
*   テーブル構造、見出し階層、図表の位置情報を高精度で抽出
*   段階的に導入: Phase 1 はテキスト PDF、Phase 2 でスキャン PDF / 画像対応

### 課題 2: AI が読みやすい仕様書フォーマットの最適解
**問題:** 規格（IEEE / JIS / Arc42 など）やフォーマット（JSON / Markdown / YAML）を複合化した方がよいかもしれない。

**現状の対応:**
*   IEEE 29148 (SRS) + Arc42 (アーキテクチャ) + C4 Model (構造図) を統合
*   10 セクション・28 リーフの SPEC_TEMPLATE に標準化済み
*   出力形式は Markdown (本文) + Mermaid (図表) + JSON (メタデータ)

**今後の改善案:**
*   現状の Markdown ベースは AI の読み書きに最適解に近い（トークン効率が良い）
*   Front Matter (YAML) + Markdown 本文 + Mermaid 図 の 3 層構造が実用的
*   プロジェクト種別ごとのテンプレートバリエーション追加（Web API 向け / 組込み向けなど）
*   出力フォーマットの切替機能: Markdown → Word / PDF / Confluence への逆変換

### 課題 3: リポジトリ解析から仕様書への変換ロジック
**問題:** リポジトリを解析して仕様書に変換するロジックの正解がない。

**現状の対応:**
*   リポジトリのファイルツリー + 主要ファイルの内容を Claude API に送信
*   プロンプトで 28 セクション ID を指定し、各セクションに分類された仕様書を生成
*   spec-classifier.ts で AI 出力を SPEC_TEMPLATE にマッピング

**今後の改善案:**
*   段階的解析: まず全体構造を把握 → 各モジュールを深掘り（2-pass 方式）
*   AST (抽象構文木) 解析: TypeScript / Java / Python のパーサーでクラス図・依存関係を自動抽出
*   テスト仕様の逆引き: テストコードから仕様を推定（テストが仕様の source of truth）
*   差分解析: 全ファイルではなく変更ファイルのみ解析（コスト・速度の最適化）

### 課題 4: リポジトリ保存システムの選択
**問題:** GitHub / GitLab を借りた方がよいか、GitLab を Fork してオリジナルを開発した方がよいか。
**推奨:** GitHub / GitLab をそのまま活用

| 選択肢 | メリット | デメリット |
| :--- | :--- | :--- |
| GitHub をそのまま活用 (推奨) | 開発コスト最小、エコシステム活用、ユーザーの既存環境と統合 | GitHub に依存 |
| GitLab を Fork | 完全カスタマイズ可能 | 保守コスト膨大、セキュリティパッチ追従が困難 |
| 独自 Git サーバー | 完全制御 | 車輪の再発明、非現実的 |

**理由:**
*   Re:Spec の本質価値は「仕様書の変換・管理・同期」であり、Git ホスティング自体ではない
*   GitHub API / Webhook で十分な連携が可能（現在も Octokit で実装済み）
*   ユーザーの既存 GitHub 環境に自然に統合できる
*   将来的に GitLab / Bitbucket 対応を追加すればプラットフォーム非依存になる

### 課題 5: 技術の進歩への対応
**問題:** 技術の進歩が速すぎる。仕様コンバータ、仕様書管理プラットフォーム、リポジトリ保存システムの 3 機能をそれぞれ改善開発する必要がある。

**現状の対応:**
*   マイクロサービス設計で疎結合化済み（Main App + Converter Service）
*   Azure Storage Queue で非同期通信（サービス間の独立性確保）

**今後の改善案:**
*   AI モデルの差し替え容易性: Claude API のラッパー ( claude-client.ts ) を抽象化し、GPT5.2 / Gemini3 等への切替を容易に
*   パーサーのプラグイン化: ファイル種別ごとのパーサーを独立モジュール化（新形式追加が容易）
*   テンプレートのバージョニング: SPEC_TEMPLATE にバージョンを付与し、フォーマット変更時の移行パスを確保
*   機能フラグ: 新機能を段階的にリリースするためのフィーチャーフラグ導入

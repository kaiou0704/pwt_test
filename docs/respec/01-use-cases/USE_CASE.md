# 統合仕様書：Re:Spec ユースケース仕様

## 1. ユースケース概要
本仕様書は、Re:Specが提供する主要なユースケース、課題解決策、およびKPIを定義するものである。なお、既存の「京都春の観光モデルコース」に関する情報は、本ドキュメントの目的（Re:Specの機能定義）に基づき、参考情報として保持する。

## 2. ユースケース一覧と解決策の概要

| ユースケース | 顧客の課題 | Re:Specの解決策 | KPI |
| :--- | :--- | :--- | :--- |
| UC1 SIer受託 | 仕様書の手戻り | Word In/Out + AIインスペクション | 手戻り70〜85%削減 |
| UC2 SaaS自社開発 | 仕様書が存在しない | コード→仕様 逆生成 | オンボ3〜5日に短縮 |
| UC3 テストBPO | 人月ビジネス縮小 | 仕様書品質レビュー | 新売上 ¥200K〜/件 |
| UC4 大企業内製 | 速度と品質の板挟み | 8観点自動検査 + JSON証跡 | 監査準備90%削減 |
| UC5 金融規制 | 監査準備コスト | コード→仕様逆生成 | 年間 ¥30〜50M削減 |

| UC7 Notion PRD | PRDの陳腐化 | Sync Engine + AIインスペクション | PRDスコア 68→90点 |

## 3. ユースケース詳細

### UC1：SIer受託開発
*   **ターゲット**: 大手SIer（PM/SE 50〜200名規模）
*   **業務フロー**: 1. 顧客のWord仕様書をインポート、2. AI Inspectorが8観点で自動検査、3. 指摘箇所をWordコメントで返却、4. Sync Engineでコードとの乖離を常時監視

### UC2：SaaS自社プロダクト
*   **ターゲット**: SaaS企業（開発チーム 5〜50名）
*   **業務フロー**: 1. GitHubリポジトリ接続、AIが仕様書を逆生成、2. Notion PRDをSync Engineで接続、3. AI Inspectorで品質スコアリング、4. コード変更のたびに仕様書が自動更新

### UC3：テストBPO上流シフト
*   **ターゲット**: テスト設計チーム（10〜50名）
*   **業務フロー**: 1. 顧客の仕様書を投入、2. AIが客観的に品質スコアリング、3. 曖昧さ・矛盾を顧客にフィードバック、4. 「仕様書品質レビュー」を有償サービスとして提供

### UC4：大企業内製チーム
*   **ターゲット**: 大企業の内製開発チーム（50〜200名、Agile/Scrum開発）
*   **業務フロー**: 1. GitHub EnterpriseをSync Engineに接続、2. AI InspectorがPR連動で仕様書を自動検査、3. 品質スコア+証跡を自動記録

### UC5：金融規制対応
*   **ターゲット**: メガバンクIT部門（基幹系システム更改）
*   **業務フロー**: 1. 全仕様書を一括インポート（Word/PDF）、2. 8観点AIインスペクションを定期自動実行、3. 品質スコア+証跡をJSON記録、4. 監査レポートをワンクリックPDF出力

### UC6：レガシー仕様書救出
*   **ターゲット**: 大企業の情報システム部門（10〜20年稼働の基幹系システム）
*   **業務フロー**: 1. 散在するWord/PDF/Excelを一括インポート、2. 現行コードをリポジトリ解析、AIが仕様逆生成、3. 既存仕様 vs 逆生成仕様のDiff表示、4. マイグレーション計画の基礎資料を生成

### UC7：Notion PRDユーザー
*   **ターゲット**: SaaS企業・スタートアップのPM/PdMチーム
*   **業務フロー**: 1. Notion APIでPRDを自動取得、2. AI Inspectorで品質スコアリング、3. コード変更を検知、Notion PRDに「要更新」通知、4. PRD → SPEC_template自動変換

## 4. 参考情報：京都春の観光モデルコース
*   **移動手段**: 2日目の嵐山への移動には阪急電車を使用すること。
*   **宿泊施設**: 宿泊先は「アゴーラ 京都烏丸」を指定する。
*   **観光対象**: 歴史的建造物（清水寺、天龍寺）、自然景観（白川沿いの桜、渡月橋、竹林の小径、天龍寺庭園）、街並み（産寧坂、二寧坂、祇園の石畳）

仕様書を起点に、3つの構造変化が同時に起きている

## 01 仕様駆動開発（SDD）の爆発
* Copilot/Claude Code/Kiroで「仕様→コード自動生成」が主流に
* コード生成AIの精度は仕様書の品質に完全に依存する
* 開発は速くなったが、品質管理部門が悲鳴を上げている
* 仕様書が開発プロセスの「起点」になる時代が来た

## 02 大企業の内製化シフト
* SIer委託→内製化で「仕様書を書くSIer」がいなくなった
* Notionで書いたPRDが放置される構造的問題
* NotionのPRDでは監査証跡が残らない

## 03 規制・監査の厳格化
* SOC2/ISMS/ISO 26262/金融庁検査の要件が厳格化
* 「仕様書と実装の整合性証明」が監査必須項目に

そんな重要な仕様書が、いま壊れている

仕様書が開発の「起点」になったのに、その品質を誰も管理していない

* **73%** のプロジェクトで仕様書とコードが乖離
* **¥15M / 案件** の手戻りコスト（大規模SI）
* **60%** のレビュー欠陥が見逃される
* **4週間** 新メンバーのオンボーディング期間

* Notion PRDも3ヶ月で陳腐化。実装が進むほど仕様書は取り残される。
* 結合テストで発覚する仕様齟齬。設計フェーズの曖昧さが根本原因。
* 人間のインスペクションは疲労と属人性に依存。観点の漏れが品質リスクを生む。
* 仕様書が信用できないため、コードを読むしかない。暗黙知が組織のボトルネックになる。

AI Converter — ドキュメント変換エンジン

あらゆる形式の仕様書を構造化し、コードから仕様書を逆生成する

### 順方向: ドキュメント → 構造化仕様書
* **Word (.docx)**: テーブル・見出し・箇条書き構造を保持して変換
* **PDF（OCR対応）**: スキャンPDFも構造を解析して変換
* **Excel (.xlsx)**: マトリクス形式の要件表をMarkdownテーブルに
* **Notion PRD**: Notion APIで取得し、構造化Markdownに変換

### 逆方向: コード → 仕様書
* **リポジトリ解析**: ファイル構造・依存関係・エントリポイントを自動特定
* **コード意図推論**: 関数・クラス・APIの目的をAIが自然言語で記述
* **SPEC_template生成**: 機能仕様・IF仕様・制約を構造化テンプレートに
* **Diff表示**: 既存仕様書がある場合、逆生成仕様との差分を可視化

* **DeepWikiとの違い**: DeepWikiは「読み物Wiki」を生成。Converterは「AIが開発を駆動するSPEC_template」を生成。
* PMはWordで書き続ける。SaaS PMはNotionで書き続ける。開発者はコードだけ書く。Converterが「形式の壁」を消す。

AI Inspector — 仕様書インスペクションエンジン

IEEE 1028 / Fagan Inspectionの自動化。人間のレビュアーの最強アシスタント。

| 観点 | 精度 | 内容 |
| :--- | :--- | :--- |
| 01 曖昧さ検出 | ◎ 95%+ | 「適切な」「必要に応じて」等の曖昧表現を検出 |
| 02 テスト可能性 | ◎ 90%+ | 定量的判定基準の有無を検査 |
| 03 整合性 | ○ 75-85% | 仕様書内の矛盾・記述の不一致を検出 |
| 04 網羅性 | ◎ 85%+ | 正常系/異常系/境界値の記述漏れを検出 |
| 05 追跡可能性 | ○ 70-80% | 要件ID・上位要件との紐付けの有無を検査 |
| 06 用語一意性 | ◎ 95%+ | 同一概念の用語揺れを検出 |
| 07 実現可能性 | △ 40-60% | 技術的に実装困難な要件をフラグ |
| 08 優先度分類 | ○ 形式のみ | MoSCoWの記載有無と形式チェック |

* **インスペクション出力**:
    * 品質スコア: 0-100点の定量スコアリング
    * 指摘リスト: 観点別に分類された改善提案の一覧
    * 監査証跡JSON: いつ・何を・どの観点で検査したかの完全記録
    * 改善提案: AIによる具体的な修正文案の自動生成

Sync & Integration Engine — あらゆるPMツールと開発ツールを接続する

* **GitHub連携 — コード同期**:
    1. 検知: PR/マージのWebhookで変更を即座に検知
    2. 解析: AST/セマンティクスレベルで仕様書との差分比較
    3. 分類: 重大/軽微/意図的に自動分類
    4. 実行: 仕様書更新PRを自動作成。レビュアーは承認するだけ
* **Notion連携 — PRD同期**:
    1. 取込: Notion APIでPRDを取得し構造化Markdownに変換
    2. 検査: AI Inspectorで品質スコアリング → Notionにコメント返却
    3. 同期: コード変更→Notion PRDに「要更新」通知+ステータス更新
    4. 変換: PRD → SPEC_template自動変換。AIエージェントが即消費可能

Re:Specが実現する仕様書ライフサイクル

* **INPUT**: Notion PRD / Word / PDF / GitHub Repo
* **プロセス**:
    1. 変換: Notion PRD/Word/コード → 構造化
    2. 検査: 8観点AIインスペクション + 品質スコア
    3. 同期: コード変更検知 → 仕様書自動更新
    4. 通知: Notion/Slack/Teamsに乖離アラート
* **OUTPUT**: Word / PDF / SPEC_template / 監査証跡JSON / Notionステータス更新

* **Notion連携がもたらす戦略的優位性**:
    * フリーミアムの入口: Notion PRD品質スコアリングを無料で提供→有料版への転換
    * PMを味方にする: 「ツールを変えろ」と言わない唯一のSpec管理プラットフォーム
    * TAMの劇的拡大: Word企業+Notion企業の両方をカバー
    * データの堀: Notion+GitHub+Word全経由の仕様書データが蓄積される
